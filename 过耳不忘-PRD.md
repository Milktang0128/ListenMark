## 1. 执行摘要

过耳不忘是一款 macOS 原生工具：在任意应用里**划词选中文本**，即可对它执行一组**语音化的 AI 动作**（朗读 / 解释 / 翻译 / 提炼 / 背景），结果**全部用语音念出**，并把**每一次交互自动留档**——可搜索、可回看、可重听。它面向"需要用耳朵把内容听懂、且会回头复习"的知识阅读者，差异点在于豆包等只朗读不留存，而过耳不忘把"听"沉淀成一份会越用越厚的个人记忆。

---

## 2. 背景与脉络

- **机会**：划词 TTS 已被豆包等验证，但它们"念完即弃"。真正空白的是「**自动记住 + 可回看**」这一层。
- **核心洞察**：产品名承诺的是"记住"，而单纯朗读只交付了"记录"。护城河不是已商品化的 TTS，而是**日积月累的个人语料 + 你能拿它做什么**；TTS 是获客钩子，记忆层才是产品本体。
- **两条产品公理**（不可动摇）：
  - **① 全部用语音**：任何动作的结果都先以语音抵达耳朵。
  - **② 全部留档**：每次交互＝原文＋动作＋来源＋AI 回应，自动沉淀。
- **定位边界**：voice-first 把产品框定为**理解型**工具（帮我听懂），天然排除改写 / 写作类**生产型**动作——这是有意的聚焦，不是缺失。
- **最大风险假设**：V1——用户真的会回看归档内容。本 PRD 不靠前置实验验证它，而是把它转化为上线后的**北极星指标**持续监控（见 §3、§8）。

---

## 3. 目标与成功指标

**目标（Goals）**
1. 让"划词 → 选动作 → 听结果 → 自动留档"成为一个**亚秒级、零摩擦**的闭环。
2. 让档案**值得且方便回看**——回看/重听是一等公民，不是埋藏功能。
3. 用真实使用数据**验证 V1（会回看）**，作为是否加大投入的 go/no-go 依据。

**非目标（Non-Goals）**
1. **不做改写 / 润色 / 正式化等生产型动作**——输出是用来复制的文字，违背"全部用语音"，留给写作工具。
2. **不做整页沉浸式翻译**——我们是段落级、语音化、留档，不与 immersive translate 正面竞争。
3. **不做音视频 / 播客转写**（MVP）。
4. **不做 Windows / 移动端 / 多人协作分享**（MVP）。
5. **不做跨端同步**（MVP，列入 P2）。

**成功指标（Success Metrics）**

## 4. 目标用户与细分

**主滩头（提案，待 §7 确认）**：母语非英语、需要大量阅读**英文 / 专业 / 难懂内容**的知识工作者与研究生。
- 为什么是他们：朗读 + 解释 + 翻译 + 背景 这套组合在"读硬内容"时价值最大；且他们有强烈的**回看 / 复习**动机（研究、参考、学习），天然契合 V1。
- 现有替代：复制到豆包/ChatGPT 念 → 散落无留存；划词翻译工具 → 只翻不留、无语音理解；微信读书/Readwise → 只在自家内容里、不跨应用。

**次级人群**：中文深度阅读者（公众号 / 长文 / 书）、视障与护眼需求者。

## 5. 用户故事与需求

### P0 — Must Have（核心闭环 + 记忆 + 隐私）

| # | 用户故事 | 验收标准 |
|---|---|---|
| P0-1 | 作为用户，我在**任意 macOS 应用**选中文本后能唤起过耳不忘 | 支持 Safari/Chrome/PDF 阅读器/原生 App/主流 Electron（VS Code、飞书）；热键或自动浮层唤起；Accessibility 取词失败时回退到剪贴板；触发到响应 <1.5s |
| P0-2 | 作为用户，我能**一键朗读**选中原文 | 默认浮"朗读"键，单击即读；streaming 边合成边播；可暂停 / 停止 |
| P0-3 | 作为用户，我能对选中文本选 **解释 / 翻译 / 提炼 / 背景**，并听到结果 | 浮层可展开菜单；LLM streaming → TTS 边生成边念（句子级切分）；每个动作有合理默认值（翻译默认目标语、解释默认深度） |
| P0-4 | 作为用户，我的每次交互都被**自动留档**，无需手动保存 | 存原文、动作类型、来源（App / 标题 / URL）、时间、AI 回应 transcript、音色；写入失败自动重试不丢 |
| P0-5 | 作为用户，我能**打开档案、按时间浏览、全文搜索、一键重听**任意条目 | FTS 全文检索覆盖原文与 AI 回应；可按动作 / 来源筛选；重听走缓存或按需重新 TTS |
| P0-6 | 作为用户，我能**信任**"记录我读过的一切"不外泄 | 数据默认仅本地存储；提供"暂停记录"开关、按 App 排除、单条删除；明确告知哪些文本会发往云端 LLM/TTS |

### P1 — Should Have

| # | 用户故事 | 验收标准 |
|---|---|---|
| P1-1 | 作为用户，我每天能看到一个**今日回响**复习列表 | 入口在档案页，**不推送、不打扰**；可一键连续重听；选取策略（随机 + 时间间隔）可配置 |

## 6. 方案概述

**形态**：Swift + SwiftUI 菜单栏 App（macOS 原生）。

**关键设计决定**
1. **默认单键、进阶展开**：划词后默认只浮"朗读"键，hover / 右滑 / 快捷键再展开其余动作——默认即用，不每次逼用户选。
2. **Streaming 全链路**：解释 / 翻译 = LLM 生成 + TTS 两段延迟，必须边收 token 边按句合成边播放，否则等到崩。朗读动作跳过 LLM、直接 TTS 原文。
3. **本地优先留档**：SQLite + FTS5 全文索引；音频按需重新合成（默认不缓存大文件），离线 / 保原音色再考虑缓存。

**技术链路**
- **取词**：Accessibility API（`AXSelectedText`）优先 → 模拟复制读剪贴板回退；浮层用透明 `NSPanel`。
- **处理**：动作 → prompt 模板 → LLM（streaming）；朗读绕过 LLM。
- **语音**：streaming TTS provider，句子级切分边合成边播。
- **存储**：SQLite + FTS5；Markdown 导出；可选音频缓存。
- **隐私**：本地优先；明确云调用边界；暂停 / 排除 / 删除。

## 8. 时间线与分期

- **Phase 0 · 技术突刺（1–2 周）**：取词覆盖率 + 延迟实测（发现计划 E3）；选定 LLM/TTS 供应商。**门槛**：≥80% 常用应用稳定取词、首字 <1.5s，否则形态降级为浏览器扩展。
- **Phase 1 · MVP（4–6 周）**：交付全部 P0（核心闭环 + 留档 + 档案视图 + 隐私）。小范围内测（含作者本人为零号用户）。
- **Phase 2 · 记忆与陪听（3–4 周）**：交付 P1（今日回响、听读队列、音色、导出）。扩大 Beta，**重点监控周回看率以验证 V1**。
- **Phase 3 · 扩展**：P2 候选动作（追问 / 续讲 / 查证）+ iOS 续听。

**Go / No-Go**：Phase 2 周回看率 ≥40% → 加大投入做 P2 与商业化；<15% → V1 证伪，回头重估方向 B（陪听）是否才是真需求。
