# 过耳不忘 · 产品发现计划

**日期**：2026-06-13
**产品阶段**：新产品（New Product，需求未被验证）
**发现主线**：在已被豆包等验证的「划词 TTS」之上，「自动归档 + 可回看」这一层，能否成为用户愿意长期使用、甚至付费的真正理由？
**本轮决策目的**：规划一个能上手的 MVP

> **2026-06-13 更新**：用户决定暂不做验证实验（第四节保留备查），转入产品定义；并新增核心需求——划词后可选「朗读 / 解释 / 翻译 / …」多种动作，**全部用语音、全部留档**（详见〇、动作集）。

---

## 一句话产品雏形

> **过耳不忘**：电脑上划词选中任何文本 → 选一个**动作**（朗读 / 解释 / 翻译 / 提炼 / 背景 / …）→ 大模型处理 → **结果用语音抵达耳朵** → **整段交互自动进档案，可搜索、可回看、可重听**。
>
> 两条产品公理：**① 全部用语音**（任何动作的结果都先念出来）；**② 全部留档**（每次交互＝原文＋动作＋来源＋AI 回应，自动沉淀）。

**核心洞察（贯穿全程）**
- **承诺 vs 机制的鸿沟**："过耳不忘"承诺的是**记住**，而当前描述只交付了**记录**。从"记录"走到"记住"（复习 / 回响 / 再触达）那一步，才是相对豆包真正立得住的护城河。
- **护城河不是 TTS**（已商品化），而是「日积月累的个人语料 + 你能拿它做什么」。**TTS 是获客钩子，文档 / 记忆层才是产品本体。**
- **最大风险点**：A（坐下专注复习）与 B（分心免手听）是两种相反的注意力状态，缝合处即风险。MVP 先押 A 的"记忆"价值。

## 〇、动作集（语音化 AI 动词）

划词浮层不再只有"朗读"，而是一组**动词**。所有动作共享两条公理：结果用语音输出、整段交互留档。

**v1 菜单（语音原生）**

| 动作 | 触发心智 | 语音输出 | 留档内容 |
|---|---|---|---|
| 朗读 | 把这段念给我听 | 念原文 | 原文 |
| 解释 | 这段什么意思 / 讲通俗点 | 念 AI 解释 | 原文＋解释 |
| 翻译 | 翻成中 / 英… | 念译文 | 原文＋译文 |
| 提炼 | 一句话说重点 | 念摘要 | 原文＋摘要 |
| 背景 | 这是谁 / 什么梗，补点背景 | 念背景科普 | 原文＋背景 |

**候选动作（按需加）**：追问 / 对话（语音问答，全程留档）· 接着读 / 续讲 · 纠错 / 查证 · 换个说法 / 给例子 · 音色与语速切换。

**不纳入（会破坏"全部用语音"）**：改写 / 润色 / 正式化——输出是用来复制的文字，语音是错的载体；留给写作工具，不稀释"用耳朵理解世界"的定位。

**两个判断**
- "全部用语音"是有用的约束：把产品框定为**理解型**工具，天然排除生产 / 写作型动作，MVP 不发散。
- 档案因此更值得回看：从"我读过的段落"升级为"我遇到的内容＋我当时得到的解释 / 译文"，后者更可能被回头看——**正面加固最大风险假设 V1（会回看）**。

**三个设计决定**
1. 菜单 UX：默认只浮"朗读"，hover / 右滑展开其余动作（默认即用、进阶可选）。
2. 延迟：解释 / 翻译＝LLM 生成＋TTS 两段延迟，必须**边生成边念（streaming TTS）**。
3. 留档：存文字 transcript（原文＋AI 回应），音频按需重新合成；离线 / 保原音色再考虑缓存。

## 二、选中方向（收敛）

| 方向 | 价值主张 | 早期人群 | 理由 |
|---|---|---|---|
| **A 听读知识库** | 记住 & 召回：档案可搜索、可复习、会主动回响 | 知识工作者、读者、学生 | 最直接兑现"过耳不忘"四个字，是相对豆包唯一立得住的差异 |
| **B 全场景陪听** | 免手连续听、护眼、多任务 | 重度阅读者、眼疲劳者、通勤族 | 提高朗读环节的使用频率，喂大档案语料 |

**MVP 优先押 A 的"记忆"价值**；B 的连续听作为提高使用频率的辅助能力，跨端同步延后。

## 四、验证实验

| # | 验证假设 | 方法 | 成功标准 | 工作量 | 周期 |
|---|---|---|---|---|---|
| **E1** | V1（+顺带 V2） | 协和式 pretotype：热键 glue + 真人跑 2 周 | 种子用户中 ≥40% 在第 2 周仍产生 ≥1 次主动回看 / 重听 | 中（1–2 天搭建） | 2 周 |
| **E2** | V1 的存量证据 | The Mom Test 行为访谈 ×5–7 + 外部留存数据 | ≥3 人有稳定回看自存内容的行为 | 低 | 1–2 天 |
| **E3** | F1 + F2 | macOS 可行性突刺（spike） | ≥80% 常用应用稳定捕获 & 首字延迟 <1.5s | 中 | 2–3 天 |

### 实验细节

**E1 — 协和式 pretotype（"假装已经做好了"）**
- XYZ 假设：至少 **40%（X）** 的种子重度阅读者，在用满 **2 周（Z）** 后，会在累积档案里产生 **≥1 次主动回看 / 重听（Y）**。
- 搭建：用 Hammerspoon / Automator 写一个 ~30 行热键脚本：选中文本 → 调 TTS API 播放 → 把"原文 + 来源 App + 时间"append 进一个 Markdown 文件（直接放 Obsidian）。**不做任何 UI 打磨**。
- 招募：5–8 名重度阅读者（含你自己——你是最佳零号用户）。
- 度量：回看率（打开 / 搜索 / 重听档案的人数比例与频次）、被回看的段落特征、第 2 周留存。
- 决策线：≥40% 周回看 → V1 成立，进 MVP；<15% → V1 证伪，回到方向选择（也许 B 比 A 更真），重估。

**E2 — Mom Test 行为访谈（先于 E1，给它定校准）**
- 不问"你会不会用"，只问过去行为：你上次回看自己存的高亮 / 摘录是什么时候？怎么找到的？为什么当初要存？
- 外部证据：看 Readwise / 微信读书 / flomo 的复习留存——别人靠"回看高亮"做成了生意，是 V1 的存量信号。
- 1–2 天即可完成。

**E3 — macOS 可行性突刺（因选了原生路线，升为开工前必验）**
- 时间盒 2–3 天，用 Accessibility API / NSPasteboard / CGEventTap 做最小 spike，测：能否在 Safari / Chrome / PDF / 原生 App / Electron（VS Code、飞书）里稳定取到选中文本？首字出声延迟多少？
- 决策线：≥80% 常用应用稳定捕获 & 延迟 <1.5s → 原生路线可行；否则降级为"浏览器扩展先行"或"独立 App + 粘贴导入"。

## 五、MVP 范围（验证通过后）

**做（v0.1）**
- macOS 菜单栏 App + 全局热键 / 划词浮层，浮层带**动作菜单**（朗读 / 解释 / 翻译 / 提炼 / 背景）
- 语音输出：所有动作结果经 streaming TTS 念出；朗读念原文，其余念 AI 回应
- 自动留档：每次交互 → 本地存储（SQLite + Markdown 导出），含原文、动作、来源 App、时间、AI 回应 transcript、音色
- 档案视图：时间线 + 全文搜索（含 AI 回应）+ 一键重听
- 轻量「今日回响」：每天把 N 条旧条目汇成被动列表（不推送、不打扰）——A 方向最小内核
- 听读队列：连续免手播放多段 / 多条回应（B 方向最小内核）

**不做（v2+）**
- 跨端同步（F3）、多音色 / 情感 / 角色朗读（C）、对段落追问（D9）、付费墙——全部延后

---

## 六、发现时间线

- **第 1 周**：E2 访谈（d1–2）+ E3 spike（d1–3）并行；E1 脚本搭建（d1–2）后开跑
- **第 2 周**：E1 持续运行；周中看回看信号是否冒头
- **第 3 周**：汇总 E1 数据 + E2/E3 结论 → 决策

---

## 七、决策框架

- **E1 成功（≥40% 回看）且 E3 可行** → 建 macOS 原生 MVP，按第五节范围
- **E1 失败（<15% 回看）** → 暂停 A，重测 B（陪听）是否更真，或转「独立 App + 粘贴」先验记忆层
- **E3 失败** → 保留 A 价值假设，但 MVP 形态降级为浏览器扩展先行
- **E2 已显示无人回看高亮** → 提前给 E1 降权，优先重估方向 B
